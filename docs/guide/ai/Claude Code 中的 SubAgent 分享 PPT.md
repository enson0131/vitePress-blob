## 02. 今天要解决什么问题？

当我们把越来越多任务交给同一个 Agent 时，通常会出现三个问题：

- 上下文越来越乱，真正重要的信息被稀释
- 权限越来越大，审查类任务也可能误改代码
- 职责越来越杂，一个 Agent 同时做分析、修复、审查、汇总

一句话总结：

**不是模型突然变笨了，而是单一 Agent 的工程边界到了。**

---

## 03. 一个典型场景

想象这样一个连续操作：

1. 让 Claude Code 跑测试
2. 让它分析错误日志
3. 让它理解代码结构
4. 再让它修一个 bug

问题在于：

- 测试日志几百行
- 错误输出几百行
- 代码分析又是一大段
- 最终真正要修的问题，反而被埋在中间过程里

> 演讲提示：这一页重点让听众共鸣。可以直接说，很多人感受到的“AI 越用越健忘”，本质上是上下文污染，不是模型退化。

---

## 04. 什么是 SubAgent？

SubAgent 可以理解为一个“专职小助手”：

- 拥有自己的上下文窗口
- 只负责某一类明确任务
- 可以被限制为特定工具权限
- 完成后把摘要结果返回给主 Agent

所以它不是为了“多搞几个 Agent”，而是为了：

**把复杂任务拆开，把中间过程隔离，把主 Agent 从噪声里解放出来。**

---

## 05. SubAgent 的三大价值

### 1. 隔离

- 把高噪声过程隔离出去
- 主 Agent 只保留结论，不保留全部细节

### 2. 约束

- 给不同任务配置不同权限
- 审查任务不必拥有写权限

### 3. 复用

- 子代理可以沉淀成文件
- 可以纳入版本控制，团队共享

---

## 06. 什么时候适合用 SubAgent？

以下四类任务最适合：

### 高噪声任务

- 跑测试
- 分析错误日志
- 扫描大规模输出

### 需要权限隔离的任务

- Code Review
- 安全审计
- 只读数据库查询

### 可并行拆分的任务

- 多模块探索
- 多方向排查

### 流水线任务

- 定位问题
- 分析问题
- 修复问题
- 验证问题

---

## 07. 什么时候不适合用？

如果任务高度依赖人与 Agent 的持续对话，就不适合过度拆分。

例如：

- 需要用户频繁确认方向
- 需求本身还在不断变化
- 目标并不清晰，只是在探索思路

一句话判断：

**如果关键在“持续沟通”，而不是“独立执行”，就不要强行拆成 SubAgent。**

---

## 08. 子代理文件长什么样？

一个 SubAgent 文件通常放在 `.claude/subagents/` 目录下。

它由两部分组成：

- YAML frontmatter：描述名称、权限、模型、模式
- Prompt 模板：定义角色、执行步骤、输出格式

示例：

```markdown
---
name: code-reviewer
description: Review code changes for quality and security. Use after code is modified.
tools: Read, Grep, Glob
model: sonnet
---

你是一个代码审查专家。

当被调用时：
1. 理解变更范围
2. 检查安全问题
3. 检查代码规范
4. 输出改进建议
```

---

## 09. 最重要的配置项

### description

决定 Claude 何时调用它。

好的 description 要同时说清楚：

- 它做什么
- 什么时候用

### tools 或 disallowedTools

决定它有哪些工具权限。

### model

决定用哪个模型处理任务。

### permissionMode

决定执行时的权限策略。

> 演讲提示：这一页不需要展开每个字段的全部细节，只强调“description 决定触发，tools 决定边界，permissionMode 决定安全级别”。

---

## 10. tools 与 permissionMode 很关键

例如做 Code Review 时，我们通常希望它：

- 能读代码
- 能搜索引用
- 能查看上下文
- 不能改文件

这时就可以这样配置：

```markdown
---
name: code-reviewer
tools: Read, Grep, Glob, Bash
permissionMode: plan
---
```

重点不是它有没有 Bash，
而是：**即便有 Bash，它也仍然处于只读探索模式。**

---

## 11. hooks 解决什么问题？

如果说 tools 控制的是“可以用什么工具”，
那么 hooks 控制的就是“工具可以怎么用”。

例如数据库查询子代理：

- 允许使用 Bash
- 但只能执行只读 SQL

这比把“只能查不能改”写在 Prompt 里更可靠，
因为约束已经从文字提醒，变成了执行前校验。

---

## 12. 工作流编排：并行模式

适合场景：接手一个新项目，需要快速理解多个模块。

例如：

- 一个 SubAgent 看 Auth
- 一个 SubAgent 看 Database
- 一个 SubAgent 看 API

然后由主 Agent 统一汇总。

并行模式的前提：

**这些子任务之间没有强依赖。**

---

## 13. 工作流编排：流水线模式

适合场景：修复一个 bug。

典型流程：

1. 定位问题
2. 分析原因
3. 实施修复
4. 验证结果

这四步之间是强依赖关系，
所以更适合串行流转，而不是并行执行。

一句话：

**独立任务走并行，有依赖的任务走流水线。**

---

## 14. 普通 SubAgent 的边界

SubAgent 已经很有用了，但它有一个限制：

**子代理之间通常不能直接交流。**

这意味着：

- A 找到一个线索
- B 也找到一个线索
- 但 A 和 B 不知道彼此发现了什么

于是一些需要“交叉推理”的问题，就容易漏掉关键结论。

---

## 15. 这时就需要 Agent Team

Agent Team 解决的是：

**让多个 Agent 不只是并行干活，而是围绕同一个问题互相交流、互相挑战、共同推进。**

适合这类问题：

- 现象复杂
- 原因可能分散在多个系统
- 需要多视角交叉验证

例如排查“用户登录后偶尔会话丢失”：

- JWT 过期时间问题
- Redis session 竞态问题
- 负载均衡 sticky session 问题

这时团队协作比孤立执行更有效。

---

## 16. 从 SubAgent 到 Multi-Agent

很多团队的一个常见误区是：

**问题还没复杂到那个程度，就先把架构搞复杂了。**

更合理的演进路线通常是：

1. 单一 Agent
2. 单 Agent + Skills
3. Supervisor + 多个 SubAgent
4. 混合式 Multi-Agent

也就是说：

**多 Agent 不是起点，而是系统复杂度演进后的结果。**

---

## 17. 如何判断要不要上多 Agent？

可以用这套简单判断逻辑：

- 单一领域、工具少、上下文不大：单 Agent 就够了
- 工具很多，但仍是单领域：优先考虑 Skills
- 多领域、需要上下文隔离：考虑 SubAgent
- 多步骤状态流转：考虑 Handoff
- 多数据源并行查询：考虑 Router

一句话：

**先问边界在哪，再问要不要多 Agent。**

---

## 18. 最后的结论

请记住这三句话：

### 第一句

SubAgent 的核心不是“多”，而是“隔离”。

### 第二句

什么时候该拆，不看潮流，看任务是不是高噪声、可分工、需控权。

### 第三句

多 Agent 架构要后置，不要在问题还简单的时候过早设计复杂系统。
